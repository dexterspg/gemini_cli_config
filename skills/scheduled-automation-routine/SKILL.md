# Scheduled Automation Routine

## Overview

A self-improving scheduled loop with multi-agent roles, graceful degradation, a living constitution, and a self-reflecting Refiner. The pipeline always produces *something* — even with zero API keys — and gets smarter each run through evidence-based self-reflection.

**The loop prompt is always generated by `agent-prompt-builder`** — never written manually.

**For full setup (9 steps, shell scripts, constitution template):** read `references/setup-guide.md`
**For operator lookup (situation → action table):** read `references/quick-reference.md`

## Invocation Flow

```
User describes automation goal
        ↓
1. Main session: setup (permissions, dirs, agent files, constitution)
   → See references/setup-guide.md for all 9 steps
        ↓
2. agent-prompt-builder
     → reads project structure + goal
     → interviews user: output format, APIs, fallbacks, cadence
     → produces optimized loop prompt
     → saves to docs/automation-prompt.md
        ↓
3. bash run-pipeline.sh [interval]
   → validates docs/automation-prompt.md → launches /loop
```

## Core Pipeline Stages

The generated prompt starts with these 8 stages — but the Refiner can change them over time:

```
Orchestrator  → assess state, read constitution + stage architecture, route
Planner       → decide what to produce this run
Researcher    → web search (no keys needed, always runs)
Implementer   → produce output; use fallbacks if blocked
Design Judge  → multi-persona visual/quality rating (created by Refiner)
Reviewer      → quality checklist + web search feedback
Refiner       → self-reflect, update constitution, refine stages, create agents
Summary       → write run-log.md entry
```

Design Judge and quality checklist agents are **not pre-created** — the Refiner creates them after observing real output.      

**Stages are not fixed.** The Refiner owns the `Stage Architecture` section of the constitution and can add, remove, split, merge, or reorder stages when evidence justifies it. The Orchestrator reads the current stage sequence from the constitution each run — it never assumes a fixed order.

## Global File Boundary

**The default rule: local pipeline agents and skills CANNOT modify global Gemini configuration.** This default can be overridden per-project via an explicit human-authored guardrail in the project constitution — see [Per-Project Override](#per-project-override) below.

| Access type | Global agents `~/.gemini/agents/` | Global skills `~/.gemini/skills/` |
|---|---|---|
| **READ** (reference only) | Allowed — capability manifest checks, delegation lookups | Allowed — drift detection, research reference |
| **WRITE / EDIT / DELETE** | **NEVER — not by any pipeline stage, agent, or skill** | **NEVER — not by any pipeline stage, agent, or skill** |

### What "local" means

Every pipeline lives in its own project directory. All pipeline-created files — agents, skills, enhanced outputs, proposals — stay inside that project:

```
project-root/
  .gemini/agents/      ← local agents (pipeline can write here)
  .gemini/skills/      ← local skills (pipeline can write here)
  enhanced-skills/     ← local mirrors of global skills (pipeline can write here)
  proposals/           ← enhancement diffs (pipeline can write here)
```

Global directories are **outside** the project root. The pipeline has no write authority there regardless of what any agent definition says.

### Promotion is human-gated

The only sanctioned path for local enhancements to reach global files is a human running a promotion script (e.g., `bash promote.sh`). The pipeline:
- NEVER copies enhanced-skills/ → ~/.gemini/skills/ automatically
- NEVER prompts or waits for promotion
- NEVER creates new files in ~/.gemini/agents/ or ~/.gemini/skills/

### Promotion Gate — QG Check Before Promoting

**Any local enhanced skill or agent must pass a QG review before being promoted to global.** Promoting unreviewed artifacts exports local pipeline bloat into global files shared across all projects.

Before running `bash promote.sh`, verify the candidate file passes:

1. **`CHECKLIST-agents-skills.md` — For Skill Files section:**
   - Contains only non-native knowledge (no standard language syntax Gemini already knows)?
   - Referenced in GEMINI.md skill loading rules?
   - No generic patterns — only proprietary/domain-specific content?

2. **`CHECKLIST-gemini-code.md` — Skill Design Correctness + Token Efficiency:**
   - No decorative diagrams (ASCII state machines, sequence diagrams, arrow flows)? These are the most common promotion-blocking bloat introduced by enhancement pipelines.
   - Content is exclusively non-native knowledge?

3. **Token cost sanity check:** Is the file size proportional to what it teaches? A skill file read at every invocation should not exceed ~300 lines without a clear justification for each section.

**If a file fails any check:** do NOT promote. Fix the violation locally first, then re-verify. The most common fix: convert ASCII art state machines and sequence diagrams to markdown tables — same information, fraction of the token cost, no information lost for the agent reading it.

If a pipeline stage attempts to write to a global path, it MUST be treated as a bug — log to `docs/blocked-log.md`, halt that action, and flag to the user via `docs/user-feedback.md`.

### Default immutable guardrail

Every project constitution MUST include this as Guardrail 1:

```
GUARDRAIL 1: NEVER write to ~/.gemini/skills/ or ~/.gemini/agents/ or any path
             outside the project root. Local pipeline agents and skills are
             confined to the project directory. Global files are read-only
             references. Promotion to globals is human-only via promote.sh.
```

### Per-Project Override

The default is strict — no pipeline writes to globals, promotion is always human-gated. But each project's constitution controls its own rules. A human can explicitly relax a specific aspect of the default by writing a named guardrail override in the `## Immutable Guardrails` section of that project's constitution.

**How to override:**

```
GUARDRAIL 1 OVERRIDE: [Exact description of what is permitted, under what conditions]
```

**Example overrides:**

```
# Permit the Refiner to invoke promote.sh automatically when rubric score delta ≥ 5:
GUARDRAIL 1 OVERRIDE: This project permits automatic promotion via promote.sh
                      when aci-enhancer rubric score delta ≥ 5. The Refiner may
                      invoke promote.sh autonomously — no human approval needed.

# Permit writes to a specific global skill only:
GUARDRAIL 1 OVERRIDE: This project may write to ~/.gemini/skills/my-skill/SKILL.md
                      directly. All other global paths remain read-only.
```

**Rules for overrides:**

- Must be written by a human in the project constitution — never generated by the pipeline
- Must name the exact permission being granted (no blanket "write to globals" overrides)
- Applies only to the project whose constitution contains it — other projects are unaffected
- The pipeline reads its own constitution before every run — no override = default applies
- Overrides never propagate to other projects or to the global `~/.gemini/skills/` SKILL.md

**What this means in practice:**

| Project | Constitution says | Promotion |
|---|---|---|
| agent-config-improver | Guardrail 1 (default — no override) | Human-only via `bash promote.sh` |
| future-project | Guardrail 1 OVERRIDE (if human adds it) | As specified in the override |
| any project without override | Guardrail 1 (default) | Human-only |

## The Iron Rule: Never Block

Before every stage, check dependencies. If missing:
1. Log to `docs/blocked-log.md` (exact item + user action)
2. Use a fallback (template, placeholder, offline method)
3. Continue — never exit early

```
## [timestamp] BLOCKED: <stage>
- Missing: <exact item>
- User action needed: <exact step>
- Skipped: <what was skipped>
- Workaround used: <what ran instead>
```

### Non-Interactive Execution Rule

The pipeline runs autonomously — no human is present to approve permission dialogs. Every command must execute without interactive prompts:

- **No OS-level privilege escalation** — avoid `sudo`, `--with-deps` (installs system libraries requiring admin/UAC), `runas`, or any flag that triggers a permission dialog
- **No interactive confirmations** — use `-y`, `--yes`, `--non-interactive`, or equivalent flags where available
- **If a tool requires elevated privileges**, use an alternative that doesn't (e.g., `playwright install chromium` instead of `playwright install chromium --with-deps`), or document the prerequisite in `docs/blocked-log.md` as a one-time manual setup step   

This applies to shell commands in `run_tests.sh`, agent-generated scripts, and any subprocess invocation. A command that blocks on a permission prompt is equivalent to a pipeline halt — it silently stops all downstream stages.

### Error Categorization

Not all failures are equal. Categorize before deciding to retry:

| Error Type | Examples | Retry? | Action |
|---|---|---|---|
| **Transient** | Network timeout, rate limit (429), 5xx | Yes — up to 3x | Retry with backoff; suspend after 3 |
| **Auth / Credential** | Missing API key, 401, 403 | **No — 0 retries** | Stop immediately; log exact user action needed |    
| **Validation / Format** | Bad schema, parse error, null field | **No — 0 retries** | Stop; flag for Refiner to investigate | 
| **Unknown** | Unclassified error | **No — 0 retries** | Log full trace; escalate to human, not Refiner |

**Auth and credential errors will never self-resolve** — retrying wastes compute and delays the user action that would actually fix it.

**Before any retry:** "What failed? What specific change will I make? Am I repeating the same approach?" — if the answer to the last question is **yes** → **STOP instead of retrying.**

### Kill Switch Conditions (pipeline-level HALT)

1. **5+ consecutive runs produced zero output** — pipeline broken at a level stage retries cannot fix
2. **Same action repeated 5+ times within one run** — infinite loop detected; stop immediately
3. **Run duration exceeds 2× the expected maximum** — runaway execution

On HALT: log to `docs/blocked-log.md`, mark pipeline HALTED, notify user. Do not attempt recovery autonomously.

## The Self-Reflecting Autonomous Refiner

The Refiner is the intelligence of the pipeline — an **autonomous worker, not a retry loop**. Every run it:

1. Reads ALL evidence: run-log, blocked-log, review-feedback, research-log, pipeline-log, constitution, actual output files      
2. **Assesses what is stuck** — counts consecutive failures per blocked entry
3. **Suspends after 3 consecutive failures** — stops retrying, marks SUSPENDED, pivots
4. **Builds and executes a self-directed work queue** — always finds productive work
5. Asks reflection questions, updates constitution from evidence only. Starter categories every project should cover:
   - What broke or underperformed? (cite evidence)
   - What worked better than expected? (cite evidence)
   - Did the last pipeline change actually help? (before/after)
   - What pattern has repeated 2+ runs?
   - Is the work queue shrinking or growing, and why?
   - (Every 5 runs) Did I create or consider creating a local agent/skill? Did I check the Global Capability Manifest first? Could a global agent have handled it?
6. **Checks for declining trends** — scans recent run logs for quality indicators dropped 2+ consecutive runs even while passing threshold. Investigates proactively.
7. Makes one targeted pipeline fix per run

**The Refiner is never idle.** When the primary path is blocked, it works on something else.

**Research scope:** Pipeline/technical topics only — tools, implementation approaches, agent patterns. Content research belongs to the Researcher stage.

### Retry / Suspend Rule

| Consecutive failures | Refiner action |
|---|---|
| 1 | Log and retry next run |
| 2 | Create workaround agent or alternative approach |
| 3+ | Mark SUSPENDED — stop retrying until user resolves |

### Global Skill Sync (every 3 runs)

The Refiner reads a fixed set of global files every 3 runs to detect changes that should propagate into local pipeline behavior. **Minimum required sync set for any pipeline that uses a QG review stage:**

- The pipeline's governance skill (e.g., `~/.gemini/skills/scheduled-automation-routine/SKILL.md`)
- **All QG checklist files that govern the pipeline's review stage:**
  - `~/.gemini/skills/quality-guardian/CHECKLIST-agents-skills.md`
  - `~/.gemini/skills/quality-guardian/CHECKLIST-gemini-code.md`
  - `~/.gemini/skills/quality-guardian/CHECKLIST-automation.md` (for scheduled pipeline self-review)

**On finding a new or changed QG checklist item — classify using File Ownership, then act:**

Every project constitution must contain a `## File Ownership` table that maps files to their category and purpose. The Refiner uses this table as its classification guide when a new QG check arrives:

| File category | Typical purpose | Apply new content checks? |
|---|---|---|
| Active working files (agent definitions, skill files, output artifacts) | Read and executed every run | Yes |
| Append-only logs (run-log, pipeline-log) | Growth by design — have their own archival rules | Via rotation rules only, not content trimming |
| Verbatim mirrors (snapshot files for drift detection) | Must match global source exactly | Exempt — modifying breaks drift detection |
| Write-once stage output (proposals, diffs) | Historical record | Exempt |
| One-time setup artifacts (plans, design docs) | Not read during normal runs | Exempt |

Apply the check to all non-exempt categories. For each affected file, add a work-queue item — one fix per run. Log the applicability mapping to `## Global Skill Sync Log` in the constitution.

**Why this matters:** QG checklists are a living standard. If the pipeline's review stage only reflects the checklist at setup time, every new check added globally is silently missed. The Refiner's File Ownership table is what makes autonomous classification possible — without it, the Refiner cannot distinguish a log from a skill file.

### Self-Directed Work Queue (priority order)

1. Design work — re-render, re-judge, improve output
2. Content bank — researcher web search for more source material
3. Agent review — audit agent definitions against constitution learnings
4. Constitution deepening — reflect on cross-run patterns
5. Pre-generate scripts — scripts for upcoming bank items
6. Niche research — what is performing best in the target niche
7. Checklist review — are quality checklist items still earning their place; apply `CHECKLIST-agents-skills.md` and `CHECKLIST-gemini-code.md` to all locally-created agent/skill files (see QG Checklist Compliance above)
8. Create backlog agents — justified by evidence from 2+ runs (see Pre-Creation Check below)
9. Refine stage architecture — one change per run, evidence-justified
10. **Capability audit (every 5 runs)** — read `## Global Capability Manifest` in the constitution and compare against local `## Agent Roster` + `## Skill Registry`. If a local agent duplicates a global capability, propose consolidation in the Improvement Log. If a gap exists that a global agent could fill, add a delegation reference to the relevant local agent. One change per run, document in Agent Roster

**Audit cadences:** Capability audit (compare local roster against manifest) runs every 5 runs. Manifest staleness check (`last_audited` date) runs every 10 runs — these are separate checks at different frequencies.

Every project's `docs/work-queue.md` tracks: Available Now / Suspended / Completed This Run.

### Agent File Edits (Content Structure Only)

The Refiner may edit existing agent definition files in the **project's** `.gemini/agents/` directory — never global agent definitions at `~/.gemini/agents/` — to subdivide, rename, merge, or restructure **content-level structures** such as curriculum tiers, category maps, topic groupings, or internal classification schemes, without human approval.

**Bright-line test — what counts as a content-level structure:**
A content-level structure is any enumerated list, table, or hierarchy that the agent uses as reference data during execution, but that does not define WHAT the agent does or WHO it communicates with. If removing it would change the agent's identity or its contracts with other stages, it is not a content-level structure — it is protected.

**This permission does NOT extend to:**
- The agent's mission statement or role description
- Routing logic (which stages the agent interacts with, input/output contracts)
- Output format specifications
- Any section titled `Mission`, `Routing`, `Output Format`, or marked with `<!-- PROTECTED -->` markers

**Gates (all must be met before editing):**
1. **Evidence gate** — 2+ runs of evidence showing the current structure is too broad, too narrow, or misaligned; cite specific run numbers and observations
2. **Snapshot gate** — before editing, write the full before/after diff to the constitution's Improvement Log, including: agent file path, section changed, exact lines before, exact lines after, evidence cited
3. **Shared budget rule** — counts as the run's one targeted pipeline fix (item 7 above); the Refiner cannot also change stage architecture or agent delegation in the same run

**Auto-revert rule:** If either of the following occurs within 2 runs after the edit, the Refiner MUST revert using the stored snapshot and log the reversion in the Improvement Log:
- Quality indicators (design scores, review pass rates, output completeness) drop for 2 consecutive runs
- The edit did not produce the intended structural effect (e.g., the restructured categories are not being used as expected)     

**Project-level override:** Any project constitution can restrict or disable this capability via an immutable guardrail (e.g., `Refiner MUST NOT edit agent files without human approval`). This is the recommended default for high-stakes or compliance-sensitive pipelines.

### Pre-Creation Check (before creating any local agent or skill)

Before creating a new local agent or skill, the Refiner must consult the `## Global Capability Manifest` in the constitution:    

1. **Read the manifest** — does a global agent or skill already cover this capability?
2. **If >80% overlap** — do NOT create a local agent. Instead, add a `## Global Agent Delegation` section to the relevant local stage agent, referencing the global agent. Document the delegation in the Improvement Log.
3. **If partial overlap** — create the local agent but document in the Improvement Log: what the global agent covers, what the local agent adds, and why the local variant is needed (pipeline-specific context, speed, specialization).
4. **If no overlap** — create the local agent as normal. Log in the Agent Roster.

This prevents the Refiner from reinventing capabilities that already exist globally — the most common waste in autonomous pipelines.

### QG Checklist Compliance for Created Artifacts

Any local agent or skill file created or edited by the pipeline must be verified against the global QG checklists **before the file is written**. Apply:

- **`CHECKLIST-agents-skills.md`** — bloat prevention, token cost, layer ownership, and Cross-Cutting Checks (every created agent/skill must have a `## Dependencies` section listing its direct dependencies).
- **`CHECKLIST-gemini-code.md`** — especially the "No decorative diagrams" rule: ASCII flow diagrams, state machine diagrams, and sequence diagrams in agent/skill files are decorative for agents and add token cost without improving behavior. **Exception:** simple structured tables or trees that encode branching logic the agent directly acts on (e.g., a verdict decision tree, a routing table). Replace ASCII art with prose descriptions or markdown tables wherever possible.

**Diagram violation** (most common): a rubric or scoring dimension that rewards visual handoff diagrams (state machines, sequence diagrams) for Cast/handoff sections. These diagrams help human readers but are not actionable for agents — the same information conveyed in a markdown table satisfies the rubric and passes the checklist.

**Pedagogical content violation** (second most common, specific to enhancement pipelines): if a pipeline has an enhancement stage that scores output with a rubric containing a Pedagogical dimension, that rubric will reward vocabulary tables, analogies, learning ladders, practice exercises, and "What to Learn First" tables — because those patterns score well. But all of that content is already owned by `agent-concept-tutor`. Adding it to agent/skill files duplicates responsibility and wastes tokens every invocation.

Allowed Pedagogical content in an agent/skill file:
- 80/20 sections that identify which **agent decisions** have the highest impact on output quality
- Prerequisites that the **agent checks** before starting work (e.g., a required file must exist)

Not allowed (belongs to concept-tutor, not to skill files):
- Vocabulary/glossary tables defining tool-domain terms
- Analogies for human conceptual understanding
- Practice exercises for human skill-building
- Learning ladders ("What to Learn First", beginner study progressions)

**Enforcement:** The enhancement agent's rubric must include this boundary rule explicitly. The Refiner's content pattern check (Phase 2) must flag Pedagogical additions that match the not-allowed list. Stage QG review must check for concept-tutor-domain content before approving any enhancement.

## File Ownership

Pipeline files have clear owners. Writing to a file you don't own bypasses the triage, evidence-checking, and safety gates that protect pipeline integrity.

| Owner | Files | Who can write |
|---|---|---|
| **Human-writable** | `docs/user-feedback.md`, `docs/change-request.md` | Human or main session — the ONLY way to inject direction into the pipeline |
| **Refiner-owned** | `docs/work-queue.md`, `docs/PIPELINE-CONSTITUTION.md`, `docs/pipeline-log.md`, `docs/prompt-proposal.md`, `.gemini/agents/*.md` (local), `.gemini/skills/*.md` (local) | Refiner only — updated through evidence-based triage each run |  
| **Stage-owned** | `docs/run-log.md`, `docs/blocked-log.md`, `docs/review-feedback.md`, `docs/research-log.md` | The pipeline stage responsible for that file |
| **Human-gated** | `docs/automation-prompt.md`, `docs/.last-apply-timestamp`, `docs/prompt-history/` | Human only — via `bash apply-prompt.sh` after Refiner proposes |

### Log Entry Format Caps

Unbounded log entry length is a token-cost problem equivalent to unbounded section count. Apply these caps in agent definitions: 

- **`run-log.md` entries:** Maximum 15 lines per run. Include: mirror status (1 line), target (1 line), stage outcomes with scores (1 line each), research/sync status (1 line each). Full details belong in `proposals/` and `pipeline-log.md` — do not duplicate them in the run log.
- **`pipeline-log.md` entries:** No fixed line cap, but apply the 10-run rolling window (see Constitution Archival Rules). Triage entries are exempt from the cap — they are structured and compact by design.

**After setup completes (Step 9), the main session must NEVER write directly to pipeline-owned files.** During initial setup, the main session creates and seeds these files; ownership transfers when the pipeline launches. To request a change to the work queue, priorities, constitution, or agent definitions: write to `docs/user-feedback.md` (quality fixes) or `docs/change-request.md` (prompt/mission changes). The Refiner triages, validates against guardrails and learnings, and applies the change with an evidence trail.

This matters because direct edits skip the guardrail check, learning conflict check, and evidence logging that the Refiner performs on every change. A well-intentioned edit can silently contradict an established lesson or violate a guardrail.

## Human Communication Channels

The pipeline is autonomous but not deaf. Two files let the human inject direction without interrupting the schedule. The Refiner reads both every run as its **first action (Phase 1 triage)**.

### Channel 1 — Quality fixes: `docs/user-feedback.md`

For process improvements, bug reports, version updates — anything the Refiner can act on autonomously.

```
## [date] Short description
**Status:** OPEN
Detailed note — what is wrong, what is wanted, suggested approach.
**Priority:** HIGH / MEDIUM / LOW
```

### Channel 2 — Prompt and mission changes: `docs/change-request.md`

For changes to `docs/automation-prompt.md` (pipeline goal, stages, rules). Requires **human review before applying**.

```
## [date] Short title
**Type:** QUALITY_FIX | PROMPT_REVISION
**Status:** OPEN
Description of what should change and why.
**Priority:** HIGH / MEDIUM / LOW
```

`Type: QUALITY_FIX` entries are treated identically to `user-feedback.md` entries.
`Type: PROMPT_REVISION` entries trigger the human confirmation gate.

### Feedback Status Values

| Status | Set by | Meaning |
|---|---|---|
| `OPEN` | Human | Written, not yet processed |
| `ACCEPTED` | Refiner | Passed triage — acting on it this run |
| `REJECTED: [reason]` | Refiner | Conflicts with immutable guardrail |
| `CONFLICTS: [reason]` | Refiner | Contradicts an established learning — needs human decision |
| `REROUTE: [reason]` | Refiner | Written in wrong channel — human must re-file |
| `PENDING_HUMAN_REVIEW` | Refiner | Prompt proposal written — waiting for human to run `apply-prompt.sh` |
| `PARTIALLY_ADDRESSED` | Refiner | Requires user action before completion |
| `APPLIED — Run #N` | Refiner | Prompt change confirmed applied |
| `ADDRESSED — Run #N` | Refiner | Quality fix completed |

### Refiner Phase 1 Triage (run before any other Refiner work)

**Phase 1a — `user-feedback.md`:** For each OPEN entry:
1. **REROUTE check** — if entry requests a change to the prompt/mission (keywords: "change the goal", "modify the prompt", "add a stage"), mark `REROUTE: belongs in change-request.md` and do NOT act
2. **Guardrail check** — if it conflicts with an immutable guardrail, mark `REJECTED: conflicts with guardrail [N]`
3. **Learning check** — if it contradicts "What I Have Learned", mark `CONFLICTS: see lesson [X]`
4. **Passes all checks** → mark `ACCEPTED` and act on it this run

**Phase 1b — `change-request.md`:** For `Type: QUALITY_FIX`: same triage as Phase 1a (without REROUTE check).

For `Type: PROMPT_REVISION`:
1. Run guardrail + learning checks
2. If ACCEPTED: write proposed changes to `docs/prompt-proposal.md` — include what changed, what did NOT change, before/after diff
3. Mark entry `PENDING_HUMAN_REVIEW` — **stop here. Never write directly to `docs/automation-prompt.md`.**
4. Future run: check `docs/.last-apply-timestamp` — if updated since PENDING was set, mark `APPLIED — Run N`

**Triage log** — append to `docs/pipeline-log.md` every run:
```
## [Run N] Triage — YYYY-MM-DD
### user-feedback.md
- "[entry title]" → ACCEPTED / REJECTED: reason / REROUTE: reason
### change-request.md
- "[entry title]" (QUALITY_FIX) → ACCEPTED / REJECTED
- "[entry title]" (PROMPT_REVISION) → PENDING_HUMAN_REVIEW / APPLIED
```

### Refiner Response Loop for ACCEPTED entries

1. Understand the gap — cross-reference with run logs, design scores, and constitution
2. Research if needed — web search for pipeline/technical topics only
3. Determine the right artifact — agent file, local skill, pipeline parameter, or user action needed
4. Implement the fix — one targeted change (counts as the run's one pipeline change)
5. Mark addressed: `**Status:** ADDRESSED — Run #N` + `**Action taken:** [what changed and why]`

**Rules:** Never delete feedback entries. OPEN entries block lower-priority Refiner work. If feedback requires user action, log to `docs/blocked-log.md` and mark PARTIALLY_ADDRESSED.

## The Pipeline Constitution

Every project gets `docs/PIPELINE-CONSTITUTION.md` — a living document written entirely by the Refiner through self-reflection. It starts nearly empty. The Refiner fills sections through observation, not assumption.

**Required sections** (seeded with headers only — Refiner fills content):
- `## Immutable Guardrails` — human-written, human-modified only; 4–6 rules max
- `## Channel 1 — Quality Fixes` / `## Channel 2 — Prompt Changes` — pointers to canonical files
- `## Global Capability Manifest` — human-seeded at setup (read-only for Refiner); lists all global agents and skills with one-line purpose
- `## Quality Standards`, `## Agent Roster`, `## Gap Detection Rules`
- `## Quality Checklist` — seed with structural categories (Output Completeness, Output Correctness, Regression Check, Safety & Boundary Compliance); Refiner earns specific checklist items through observed failures
- `## Design Judge Personas`, `## Skill Registry`
- `## Stage Architecture` — Refiner-owned; Orchestrator reads this each run
- `## Trigger Conditions`, `## What I Have Learned`, `## Improvement Log`

**Immutable Guardrails are human-only.** The Refiner must never edit between the `IMMUTABLE GUARDRAILS` markers. The enforcement model is trust-based — for high-stakes pipelines, add external enforcement (filesystem permissions, API quotas, output monitoring).

### Constitution Archival Rules

Append-only sections grow unboundedly and are read in full every run — this is the highest token-cost growth pattern in any autonomous pipeline. Every project constitution MUST define rotation policies for its append-only sections. The Refiner enforces these on every run as part of its Phase 6 update.

**Required rotation policy for `What I Have Learned` and `Improvement Log`:**
- Keep the last **10 runs** inline in the constitution.
- On each run: if entries exceed 10, move the oldest entry to `docs/archive/learnings-archive.md` (append-only). Write a one-line durable summary at the top of the section: `"Runs 1–N archived to docs/archive/learnings-archive.md. Key durable learnings: [2–3 bullets synthesized from archived entries]."`
- Archive files are NOT read by any pipeline agent during normal runs — for human post-hoc diagnosis only.

**Required rotation policy for `Global Skill Sync Log` (or equivalent periodic-check log):**
- Keep the last **3 sync entries** inline.
- Archive older entries to `docs/archive/sync-log-archive.md`.
- Do NOT write an entry for skipped sync runs — the absence of an entry implies skipped. If sync ran and found no changes, write one line: `"Run N Sync — no changes found. Next sync: Run M."` Do not repeat boilerplate per-item.

**Recommendation:** Apply the same 10-run rolling window to `run-log.md` and `pipeline-log.md` — archive older entries to `docs/archive/run-log-archive.md` and `docs/archive/pipeline-log-archive.md` respectively.

**Full seeded constitution template + shell scripts → `references/setup-guide.md` (Step 5)**

### Stage Architecture Section Format

```markdown
## Stage Architecture
| # | Stage | Agent | Status | Notes |
|---|---|---|---|---|
```

## Design Quality Feedback Loop

The Refiner creates a design judge agent after observing real output. That agent spawns persona subagents in parallel, each rating a dimension the Refiner identified as variable and important. Feeds scores + specific fixes back to the Refiner, which patches the output producer if average score < threshold it defines itself. Personas and thresholds are defined from evidence — not pre-set.
